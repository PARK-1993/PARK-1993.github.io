Swift-iOS - 2021/05/27 Optional언래핑 방법정리

Unwrapping종류

  1. 강제 언래핑(Force unwrapping) Optional변수에 값이 있는지 없는지 확인하지 않고 강제로 풀어버린다. 만약 변수가 nil일 경우 런타임에러가 발생된다. 100% 값이 들어가있다고 확신할 경우에만 사용해야한다. 
    var name : String? = nil
print(name!) // runtime Error
  2. 옵셔널 체이닝(Optional Chaining) 
    var name : String? = apartment?.room?.person?.name

    위처럼 객체의 프로퍼티의 값을 가져올때, 확인하고 싶은 객체의 값이 몇단계 아래에 존재하고 각 계층의 객체들이 옵셔널일 경우, 매번 값이 있는지 없는지 확인하는 것은 어렵다. 그래서 옵셔널 체이닝이라는 개념이 나왔고, “?”을 사용해 apartment가 존재하면 room이있는지 없는지 확인하고, room이 존재하면 person, name이렇게 값을 확인해 나간다. 만약 한가지라도 nil일 경우, name에는 nil이 저장된다.

  3. 옵셔널 바인딩(Optional Binding)

    if let

    if let name = studentName{
 // name은 이 블록내에서만 사용가능
}

    위와 같은 형식으로 studentName이 nil이 아니라면 name에 studentName의 값이 할당되고 블록내의 처리를 수행한다. 아래와 같이 사용하지는 말자.. 틀린건 아니지만 너무 올드한 느낌이다.

    if studentName != nil{
 // ...
}

    guard let

    guard let name = studentName else{
 // Error 처리
}
// name을 계속 사용가능

    if let과 차이가 있다면, 새로 할당된 변수를 계속해서 사용할 수 있다. 또한 if let의 경우는 체크대상이 nil일 경우 에러 처리를 하지 않아도 되지만, guard의 경우 반드시 nil이 아닌경우 처리방법에 대해 기술 하여야 한다. 그래서 보통 nil이 들어가면 치명적인 부분에서 많이 사용한다.

병합연산자(Nil-coalescing Operator)사용

coalesce뜻이 합치다말로 한국어로하면 Nil병합연산자 라고불리는 “??”연산자에 대해 알아보자. ??은 옵셔널을 간단하게 언래핑할 수 있게 도와주는 연산자이다. ??연산자를 기준으로 왼쪽에 옵셔널을 오른쪽에는 옵셔널이 nil일 경우 대신해서 들어갈 값을 써주면된다.

var name : String? = nil
print("My name is " + name ?? "Park")

이 경우 “My name is Park”이 출력되게 된다.

암시적으로 언랩된 옵셔널(implicitly unwrapped optionals)

그대로 해석하면 암시적으로 언랩된 옵셔널이라는 것인데, 옵셔널이긴한데 보기에는 언랩핑된것 처럼 보인다라는 의미로 볼 수 있다.

var myName : String! = nil
var yourName : String? = nil

위의 myName이 IUO이다(너무길어서 IUO로 부르겠다..) IUO은 Optional처럼 옵셔널체이닝, 옵셔널바인딩, 강제언랩핑등 사용이 가능하다.

하지만 암시적으로 언랩된 옵셔널이기 때문에, 사용시에 언래핑 처리과정을 거치지 않아도 값에 접근이 가능하다.

var lastName : String! = "Park"
var firstName : String = "JoonHyun"
print(lastName + firstName)
var myName = lastName

위와 같이 lastName에 대해 언래핑과정을 거치지않아도 사용이 가능하다. 하지만 만약 lastName이 nil이라면 runtime에러로 프로그램이 다운된다.

이러한 특징을 보면 어차피 lastName이 nil인지 아닌지 확인해야되기 때문에 Optional과 별 차이 없어보이고 오히려 번갈아 쓸 경우 더 헷갈릴 수 있을거같다.

왜 사용하는 것일까?

가독성과 편리성을 높일 수 있다고 한다. 프로그램의 구조상 어떤 시점에서 특정 옵셔널에 무조건 값이 들어있다는 것이 명확한 경우, IUO를 사용하면 불필요한 언래핑 과정과 물음표를 사용하지 않아도 된다는 것이다.

하지만 미숙한 엔지니어가 사용하거나 고급엔지니어도 실수 할 가능성이 있기때문에 안전함을 제1순위로 둔다면 사용하지 않는 것이 좋다. 안전함과 편리함, 가독성을 고려하여 사용해야된다.

Sometimes it is clear from a program’s structure that an optional will always have a value, after the value is first set. In these cases, it is useful to remove the need to check and unwrap the optional’s value every time it is accessed, because it can be safely assumed to have a value all of the time. 위 글은 Swift공식문서에서 왜 IUO사용하는지에 대해 기술한 것이다.

class Person{
    let name : String!
    init(personName : String){
        name = personName
    }
}

위와같이 클래스를 정의하면 name은 Person객체가 만들어질때 무조건 정해지기 때문에 name에 nil이 들어갈것이라는 걱정 없이 사용가능하다.
