TIL - 2021/02/23 CleanCode - 함수

작게 만들어라!

  • 20줄도 길다.
  • 짧을 수록 명백해진다.
  • if/while문에 들어가는 블록은 한 줄이어야한다.
    블록안에서 함수 이름을 적절히 짓는다면, 코드를 이해하기 쉬워진다.
  • 함수의 들여쓰기 수준은 1단이나 2단을 넘어서면 안된다.

    한가지만해라!

  • 함수내에서 의미있는 이름으로 다른 함수를 추출할 수 있다면, 그 함수는 여러 작업을 하는셈이다.
  • 함수 내 모든 문장의 추상화 수준이 동일해야한다.
    추상화수준이 섞여있으면 읽는 사람이 근본 개념과 세부사항인지 구분하기 어려워 깨어진 창문처럼 함수에 세부사항을 점점 더 추가한다. ※ 추상화 수준이 높다 : 함축되어있는 내용이 많다. / 추상화 수준이 낮다 : 함축되어있는 내용이 적다.

내려가기 규칙

  • 한 함수 다음에는 추상화 수준이 한 단계낮은 함수가 온다.
    함수 정의, 구현시에 추상화 단계별로 작성해 읽기쉽게한다. ``` cpp void AddPage(PageKind kind) { switch(kind) { case PageKind::StartPage: { createStartPage(); } break; case PageKind::MainPage: { //… } break; } }

void createStartPage() { //StartPage생성 }


## switch문
- 기본적으로 switch문은 N가지를 처리하며, 이를 피하기는 쉽지않다.
- 다형성을 이용해 switch문을 저차원클래스에 숨길 수 있다.
``` cpp
Money caculatePay(Employee e)
{
    try{
        switch(e.type)
        {
        case COMMISSIONED:
            return calculateCommissionedPay(e);
        case HOURLY:
            return calculateHourlyPay(e);
        case SALARIED:
            return calculateSalariedPay(e):
        default:
            throw (e.type);
        }catch(EmployeeType e){
            printError(e);
        }
    }
}
  • 위 코드의 문제
    1. 함수가 길다.(직원 추가시 더 길어진다.)
    2. 한가지 작업만 수행하지 않는다.
    3. SRP(Single Responsibility Principle)를 위반한다. -> 코드를 변경할 이유가 많다.
    4. OCP(Open Closed Principle)를 위반한다. -> 새직원 유형을 추가할 때마다 코드를 변경하기 때문이다.

위와 같은 코드에서 switch문을 추상팩토리에 숨긴 코드가 다음과 같다.

class Employee{
    bool isPayDay();
    Money calculatePay();
    void deliverPay(Money pay);
}

class EmployeeFactory{
public:
    virtual Employee makeEmployee(EmployeeRecord r) = 0;
}

class EmployeeFactoryImpl : public EmployFactory{
public:
    Employee makeEmployee(EmployeeRecord r)
    {
        try
        {
            switch(r.type)
            {
            case COMMISSIONED:
                return new CommissionedEmployee(r);
                break;
            case HOURLY:
                return new HourlyEmployee(r);
                break;
            case SALARIED:
                return new SalariedEmployee(r);
                break;
            default:
                throw r.type;
            }
        }catch(EmployeeType e){
            printError(e);
        }
    }
}

위와 같이 다형성을 이용하여 switch문을 숨기고 보여주지않는다.

서술적인 이름을 사용하라!

코드를 읽으면서 짐작했던 기능을 각 루틴이 그대로 수행한다면 깨끗한 코드이다.

  • 길고 서술적인 이름이 짧고 어려운 이름과 길고 서술적인 주석보다 좋다.
  • 이런저런 이름을 넣어보며 시간을 들여도 괜찮다.
  • 서술적인 이름을 사용하면 설게가 뚜렷해지고 코드를 개선하기 쉬워진다.
  • 모듈내에서 함수이름은 일관성을 위해 같은 문구,명사,동사를 사용한다.
    includeSetup, includeTeardownPages, includeSetupPages등

    함수 인수

  • 이상적인 인수 개수는 0개다. 3개는 피하는 편이 좋고, 4개는 특별한 이유가 있어야 한다.(그 이상은 안됨)
  • 멤버변수로 선언해서 인수를 대체할 수 있다면 그렇게해야한다.
  • 테스트관점에서도 인수가 많으면, 그만큼 확인해야하는 항목이 늘어난다.
  • 출력인수는 입력인수보다 이해하기 어렵고, 독자가 코드를 확인해야할 확률이 높다.

많이 쓰는 단항 형식

  • 인수를 질문에 던지는 경우
    bool fileExists(“MyFile”)
  • 인수를 뭔가로 변환해 결과로 반환하는 경우
    InputStream fileOpen(“MyFile”) string형의 파일이름을 InputStream으로 변환
    입력인수를 변환해야하는 경우 return값을 설정하는 것이 출력인수를 갖는것 보다 낫다.
  • 이벤트 함수(입력인수는 있지만 출력인수는 없음)
    passwordAttemtFailedNtimes(int attempts) 이벤트함수라는 것이 명확하도록 작성

플래그 인수

  • 함수로 bool값을 넘기는 것은 대놓고 여러가지 처리를 한다고 공표하는 것이다.
    render(bool isSuite)함수는 renderForSuite(), renderForSingleTest()로 나눠야 한다.

    이항 함수

  • 두 인수값이 하나의 값을 가리키거나(Point(0,0)) 자연적인 순서의 경우 사용한다.
  • 이항 함수가 무조건 나쁘진 않지만, 위험이 따름을 인식하고 단항함수로 바꾸려는 노력이 필요하다.
    assertEquals(expected, actual)은 하나의 값도아니며, 자연적 순서도 없다.

인수 객체

  • 인수가 2-3개 필요하다면 클래스를 만들 것을 고려한다.
    Circle makeCircle(double x, double y, double radius)
    Circle makeCircle(Point(center), double radius)
    이는 표현하는 방식만 바꾼 것이 아니라, 개념적으로 도움이 된다.

인수 목록

  • 가변 인수를 가지는 함수는 단항,이항,삼항 함수로 취급 할 수 있다.
    string format(string format, Object… args)

동사와 키워드

  • 함수의 의도를 명확히 하려면, 인수와 함수의 관계가 잘 표현되는 함수이름이 필요하다.
  • 단항함수의 경우 인수가 동사/명사 쌍을 이뤄야 한다. writeField(name)
  • 순서를 표현하기위해 키워드를 추가해 표현한다.
    assertEquals -> assertExpectedEqualsActual(expected, actual)

부수 효과를 일으키지 마라!

  • 몰래 클래스 변수를 수정하거나, 함수로 넘어온 인수나 전역변수를 수정하는 일이 없도록 한다.
    checkPassword라는 함수에서 비밀번호가 틀렸을 경우 Seesion을 초기화하는 작업이 들어 가있다면, 부수적인 작업을 하는 것이다.(이름만 봐서는 부수 효과를 예측할 수 없다.)

출력 인수

  • 일반적으로 인수를 입력인술 해석하기 때문에, 사용하지 않는 것이 좋다.
    객체지향에서는 출력인수를 사용하지않고, 객체상태를 변경하거나 객체내 변수를 출력하도록 설계해야한다.

명령과 조회를 분리하라!

  • 함수는 객체 정보를 반환하거나, 객체 상태를 변경하거나 둘중하나만 해야한다.
    bool set(string attribute, string value);
    이 함수는 속성갑을 찾아 value값을 설정한 후 성공여부를 반환하는 함수이다.
    if(set("username", "unclebob"))으로 표현하면, 의미가 모호해진다.
    username이unclebob으로 설정되있다는 것을 판단한다고 생각 할 수있다.
    다음과 같이 분리한다면 의미를 명확하게 표현 할 수 있다. 
     if(attriuteExists("username"))
 {
 setAttribute("username", "uncldebob")
 }

    에러코드보다 예외를 사용하라!

  • 에러코드를 반환하는 방식은 명령/조회규칙을 미묘하게 위반하고, 여러 단계로 중첩되는 코드를 야기한다. 
    if(deletePage(page) == E_OK)
{
  if(registry.deleteReference(page.name) == E_OK)
  {
      if(configKeys.deleteKey(page.name.makeKey()) == E_OK)
      {
          // Success
      }
      else
      {
          // Error처리
      }
  }
  else
  {
      // Error처리
  }
}
else
{
  // Error처리
}

    : 위의 예처럼 에러처리를 곧바로 처리해야하기 때문에, 깊이가 깊어진다.

    try{
  deletePage(page);
  registry.deleteReference(page.name);
  configKeys.deleteKey(page.name.makeKey());
}
catch(Exception e)
{
  // 에러처리
}
    : 위 예처럼 예외처리를 넣을 경우 코드가 간단해진다.
    하지만, try/catch는 한가지의 작업이며, 다른 작업과 섞여있으면 혼란을 줄 수 있다. 
    void delete(Page page)
{
  try{
   deletePageAndAllReferences(page);
  }
  catch(Exception e)
  {
   // 에러처리
  }
}
void deletePageAndAllReferences(Page page)
{
  deletePage(page);
  registry.deleteReference(page.name);
  configKeys.deleteKey(page.name, makeKey());
}
    위의 예외처리를 분리시키는 것이 코드를 이해하기 더 쉽다.

Error Enum

  • Error Enum이 있다면 이를 변경할 때마다 이를 사용하는 클래스의 컴파일, 재배치를 해야하고, 매번 새 에러코드 Enum을 정의하는 것도 번거롭다.
  • Exception클래스를 계승해 새로 예외클래스를 생성하면 이러한 수고를 거를 수 있다.

반복하지 마라!

  • 데이터베이스의 정규형식과 객체지향프로그래밍의 상속, AOP, COP모두 어떤 면에서 중복 제거 전략이다.

구조적 프로그래밍

  • 구조적 프로그래밍은 함수클 경우 상당한 이익을 제공하며, 작을 경우에는 별 이익을 제공하지 못한다.
  • 함수를 작게 만든다면 간혹 return, break, continue를 여러차례 사용해도 괜찮다.
※ 단일 입/출구 규칙 : 모든 함수와 함수 내 모든 블록에 입구와 출구가 하나만 존재해야한다.
함수는 return문이 하나여야 한다.
루프안에서 break나 continue를 사용해선 안되며, goto는 절대로 안된다.

결론

대가 프로그래머는 시스템을 구현할 프로그램이아니라 풀어갈 이야기로 여긴다. 프로그래밍 언어라는 수단을 사용해 좀 더 풍부하고 좀 더 표현력이 강한 함수를 만들어 이야기를 풀어야한다.
