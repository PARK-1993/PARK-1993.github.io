TIL - 2021/03/07 CleanCode 객체와 자료구조

자료추상화

사용자는 제공된 추상인터페이스를 보고 자료의 핵심을 조작할 수 있어야 진정한 의미의 클래스이다.

// 구체적
public interface Vehicle{
    double getFuelTankCapacityInGallons();
    double getGallonsOfGasoline();
}

위의 경우 단지 변수의 값을 불러오는 메소드여서 멤버변수를 public으로 하는 것과 크게 다르지않다. 하지만 밑의 경우는 단순히 변수를 가져오는것이 아니며, 사용자는 어떻게해서 데이터가 나오는지 알 수 없다.

// 추상적
public interface Vehicle{
    double getPercentFuelRemaining();
}

이렇게 자료를 세세하게 공개하기 보다는 추상적인 개념으로 표현하는 편이 좋다. 개발자는 객체가 포함하는 자료를 표현할 가장 좋은 방법을 심각하게 고민해야한다. 아무생각없이 get,set함수를 추가하는것은 의미가없다.

자료/객체 비대칭

  • 자료구조 : 자료를 그대로 공개하며 별다른 함수는 제공하지 않는다.
  • 객체 : 추상화 뒤로 자료를 숨긴 채 자료를 다루는 함수만 공개한다. 자료구조와 객체의 정의는 본질적으로 상반된다.
public class Square{
    public Point topLeft;
    public double side;
}

public class Rectangle{
    public Point topLeft;
    public double height;
    public double width;
}

public class Circle{
    public Point center;
    public double radius;
}

public class Geometry{
    public final double PI = 3.141592

    public double area(Object shape) throws NoSuchShpaeException
    {
        if(shape instanceof Square){
            Square s = (Square)shape;
            return s.side * s.side;
        }

public class Square implements Shape{
    private Point topLeft;
    private double side;

    public side*side;
}

public class Rectangle implements Shape{
    private Point topLeft;
    private double height;
    private double width;

    public double area(){
        return height * width;
    }
}

위 코드는 전형적인 객체지향 형식의 코드이며, 여기서 perimeter()를 추가하려면 모든 클래스에 추가해야되며, 도형을 하나 추가한다면 다른 클래스에는 영향없이 클래스 하나를 만들면 된다. 이처럼 객체와 자료구조는 상호보완적인 특성이있다.

모든 것이 객체라 생각하지말고, 때로는 단순한 자료구조와 절차적인 코드가 가장 적합한 상황도 있다.

디미터 법칙

잘 알려진 휴리스틱으로 모듈은 자신이 조작하는 객체의 속사정을 몰라야 한다는 법칙이다. 즉 객체는 get함수로 내부 구조를 공개하면 안된다는 의미이다. 디미터법칙은 “클래스C의 메서드f는 다음과 같은 객체의 메서드만 호출해야한다”고 주장한다.

  • 클래스C
  • f가 생성한 객체
  • f인수로 넘어온 객체
  • C인스턴스 변수에 저장된 객체
final String outputDir = ctxt.getOptions().getScratchDir().getAbsolutePath();

위와 같은 코드를 train wreck이라고 부르며, 밑과 같은 코드처럼 나누는편이 낫다.

Options opts = ctxt.getOption();
File scratchDir = opts.getScratchDir();
final String outputDir = scratchDir.getAbsolutePath();

위 식은 디미터 법칙을 위반할까? ctxt, opts, scratchDIr들이 자료구조라면 위반하지 않지만, 객체라면 위반한것이다. 객체라면 내부 구조를 숨겨야 하는데, 전혀 숨겨지지않고, 위 코드의 메소드에서 ctxt, opts, scratchDIr의 내부 구조를 알아야만 절대경로를 구할 수 있기때문이다.

만약 자료구조라면 위의 경우와는 다르게 get을 쓰지않고 공개변수만 포함하게하면 디미터 법칙을 위반안 한다는 것을 바로 알 수 있다.

final String outputDir = ctxt.options.scratchDir.absolutePath;

잡종구조

절반은 객체, 절반은 자료구조인 잡종 구조는 피해야 한다. 이러한 구조가 된 이유는 프로그래머가 함수나 타입은 보호할지 공개할 지 확신하지 못해 어중간하게 내놓은 설계에 불과하다.

구조체 감추기

만약 ctxt, options, scratchDir들이 모두 객체라면, 내부구조를 감춰야한다.

ctxt.getAbsoulutePathOfScratchDirectoryOption();
ctxt.getScratchDirectoryOption().getAbsolutePath();

첫번째 방법은 ctxt객체에 공개해야 하는 메서드가 너무 많아진다. 두번째 방법은 getScratchDirectoryOption()이 자료구조를 반환한다고 가정해야한다. 두 방법 모두 꺼림찍하다. 이러할 경우 왜 절대경로가 필요한지, 전체적인 흐름을 다시 볼 필요가 있다. 다음 코드를 살펴보니 outputDir를 가지고 임시파일을 생성하고 있다. 그렇다면 이 임시파일을 생성하는 처리자체를 ctxt가 하도록 한다면?

BufferedOutputStream bos = ctxt.createScratchFileStream(classFileName);

깔끔하다! ctxt는 내부구조를 드러내지 않으며, 해당 함수는 자신이 몰라야하는 함수를 탐색할 필요가 없다.

자료 전달 객체

자료 구조체의 전형적인 형태는 공개 변수만 있고 함수가 없는 클래스이다. 이를 Data Tranfer Object, DTO라고도 한다. 데이터베이스와 통신하거나 소켓에서 받은 메세지의 구문을 분석할 때 유용하게 사용된다.

활성 레코드

DTO의 특수한 형태이며, 공개 변수가 있거나 비공개 변수에 get/set함수가 있는 자료 구조지만, 보통 save나 find와 같은 탐색 함수도 제공한다. 활성 레코드는 데이터베이스 테이블이나 다른 소스에서 자료를 직접 변환한 결과다.
